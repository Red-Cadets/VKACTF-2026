## DESKLINE

| Событие | Название | Категория | Сложность |
| :------ | ---- | ---- | ---- |
| VKACTF 2026 | Deskline | web | easy |

  
### Описание


> Автор: d4nd3lion
>
Перед нами корпоративный мессенджер Deskline. Флаг знает только администратор, но отправляет его лишь тем, кто находится у него в друзьях.


### Решение
Регистрируемся и читаем исходники. Замечаем, что обычный пользователь не может добавить `admin` в друзья, а писать ему может только `support`. При этом `support` создаётся автоматически и сразу добавляется в друзья каждому новому пользователю.

Дальше смотрим обработку сообщений. Если написать `support` сообщение вида `/report ...`, то создаётся тикет с `ticket_uid`, а сам тикет уходит в отдельный чат между `support` и `admin`.

Также в коде видно, что если отправить `support` ссылку, то бот под логином `support` реально по ней переходит. Одновременно с этим WebSocket на ручке `/ws` проверяет только наличие валидной сессии, но никак не валидирует `Origin`, то есть перед нами CSWSH.

Если посмотреть клиентский код для `/desk/thread`, то можно увидеть готовую цепочку действий: страница открывает WebSocket от имени `support`, вызывает `sync_inbox(ticket_uid)`, получает `review_thread`, затем через `watch_ticket_thread` узнаёт `chat_id` чата с админом, отправляет туда сообщение `flag`, получает ответ админа, после чего открывает обычный диалог с пользователем и пересылает туда флаг.

Остаётся только воспользоваться этой логикой. Создаём у себя простую страницу, которая делает редирект на:

`https://<host>/desk/thread?ticket_uid=<наш_ticket_uid>`

После этого:

1. регистрируем пользователя;
2. пишем `support` что-нибудь вроде `/report test`;
3. получаем `ticket_uid` из ответа;
4. отправляем `support` ссылку на нашу внешнюю страницу;
5. бот заходит на неё, редиректится на внутреннюю страницу `/desk/thread`, та от имени `support` пишет админу `flag`, получает ответ и пересылает его нам в личку.

Забираем флаг из диалога с `support`.
