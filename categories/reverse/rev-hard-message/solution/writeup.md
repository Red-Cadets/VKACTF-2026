## Message AGC

| Событие | Название | Категория |
| :------ | ---- | ---- |
| VKACTF 2026 | Сообщение | reverse |
  
### Описание

> Автор: Invisone
>
Один из спутников, транслирующих олимпиаду упал где-то под Томском. Но все еще не перестал передавать какое-то сообщение...
## Что дано

В задании находится большой `Message.exe`. При запуске он распаковывает рабочие
файлы во временный каталог:

```text
%LOCALAPPDATA%\MessageAGC\payload-<32 hex symbols>
```

Нужные для решения файлы:

```text
Message.agc.bin
Message.agc.symtab
yaAGC.exe
```

Каталог удаляется после завершения программы, поэтому его удобно забрать во
время работы `Message.exe` через Process Monitor либо скопировать до завершения
процесса.

Сам загрузчик не расшифровывает флаг. Он запускает `yaAGC`, останавливает
исполнение на метке `SNAP` и печатает значения переменных `ERRNUM` и `ERRSUB` как
`PROG` и `NOUN`.

## Дизассемблирование AGC

В `Message.agc.symtab` присутствуют символы:

```text
START
LOOP
SNAP
SEED
STEP
STATE
IDX
TABP
TABN
```

После сборки `disassemblerAGC` из Virtual AGC дизассемблируем
`Message.agc.bin`. Существенная часть программы сводится к следующей логике.
Все константы здесь записаны в восьмеричной системе, как принято в AGC:

```text
STATE = 027
STEP  = 013
MASK  = 077
IDX   = 0

loop:
    STATE  = (STATE + STEP) & MASK
    ERRNUM = (TABP[IDX] + STATE) & MASK
    ERRSUB = (TABN[IDX] + STATE + 07) & MASK
    IDX    = (IDX + 1) & MASK
```

Маска `077` оставляет шесть младших бит, поэтому все элементы `TABP` и `TABN`
лежат в диапазоне `0..63`.

`IDX` также ограничен маской `077`, то есть проходит значения `0..63`, после
чего цикл повторяется. Поэтому 128 напечатанных пар состоят из двух одинаковых
проходов по 64 элементам. Для решения достаточно первых 64 пар.

## Восстановление TABP и TABN

`PROG` и `NOUN` печатаются как восьмеричные числа. Сначала переводим их через
`int(value, 8)`, затем обращаем операции AGC:

```python
state = 0o27

for prog, noun in pairs:
    state = (state + 0o13) & 0o77
    tabp = (prog - state) & 0o77
    tabn = (noun - state - 0o07) & 0o77
```

Порядок данных определяется самим циклом: на одном значении `IDX` программа
берет сначала `TABP[IDX]`, затем `TABN[IDX]`. Поэтому исходный поток собирается
чередованием:

```text
TABP[0], TABN[0], TABP[1], TABN[1], ...
```

Начало восстановленного потока в десятичном виде:

```text
54, 21, 11, 30, 47, 4, 44, 25, 48, 36, 29, 6, ...
```

Важно не путать системы счисления: эти значения уже переведены из восьмеричного
вывода в обычные целые числа `0..63`.

## Обоснование формата потока

Если упаковать все восстановленные шестибитные значения подряд и сразу сделать
base64 decode, получится 96 байт мусора. Причина в том, что первые два значения
не относятся к base64 payload:

```text
54, 21, <payload>
```

Первое значение `54` является длиной payload в **base64-символах**, а не длиной
расшифрованного текста. Это проверяется непосредственно по данным:

1. Пропускаем первые две шестибитные ячейки.
2. Берем ровно следующие 54 значения.
3. Каждое значение находится в диапазоне `0..63` и однозначно индексирует
   стандартный base64-алфавит.
4. Длина 54 имеет остаток 2 при делении на 4, поэтому для корректного base64
   требуется ровно `==`.
5. Декодирование дает 40 байт, что согласуется с длиной итогового флага.

Второе значение `21` является полем версии/magic текущего формата. Оно не
является частью base64: если начать преобразование с него, известный префикс
флага не восстанавливается. При начале после двух полей получается согласованный
payload. Для версии `21` используется стандартный алфавит:

```text
ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/
```

Например, первые четыре значения payload:

```text
11, 30, 47, 4
```

соответствуют символам:

```text
11 -> L
30 -> e
47 -> v
 4 -> E
```

Полная строка из 54 значащих символов с восстановленным padding:

```text
LevEsZwkdG3c65/GbUU+stmTnUVRVN3xtMh0LADTg4moVCo60Pnqgw==
```

После `base64.b64decode` получаем 40 байт:

```text
2d eb c4 b1 9c 24 74 6d dc eb 9f c6 6d 45 3e b2
d9 93 9d 45 51 54 dd f1 b4 c8 74 2c 00 d3 83 89
a8 54 2a 3a d0 f9 ea 83
```

Таким образом, поля `54` и `21` не выбираются произвольно. Граница payload
подтверждается одновременно допустимым base64, корректным padding, размером
результата и последующим восстановлением известного формата флага.

## Последний XOR

После base64 данные все еще не являются текстом. Формат флага известен:

```text
vka{
```

Используем его как known plaintext и XOR-им с первыми четырьмя байтами:

```text
0x2d ^ 'v' = 0x5b =  91
0xeb ^ 'k' = 0x80 = 128
0xc4 ^ 'a' = 0xa5 = 165
0xb1 ^ '{' = 0xca = 202
```

Полученный ключевой поток:

```text
91, 128, 165, 202
```

Каждое следующее значение увеличивается на `37` по модулю `256`. Значит маска
для байта с индексом `i` имеет вид:

```text
(i * 37 + 91) & 0xff
```

XOR обратим, поэтому расшифрование выполняется той же операцией:

```python
plain[i] = encrypted[i] ^ ((i * 37 + 91) & 0xff)
```

Решение представленно в [solve.py](solve.py)

```text
vka{s0M3_CR4zy_4rCh_n0T_g0ing_2_StoP_M3}
```
